# @beyou/contracts

TypeScript types generated from the BeYou backend's SpringDoc OpenAPI 3.1.0 spec.

## What this package exposes

- `paths` — all API route types (request/response shapes per method)
- `components` — all shared schema types
- `operations` — all named operations
- `Schemas` — shorthand alias for `components['schemas']`

The canonical types are in `src/schema.d.ts`, generated by `openapi-typescript` from the committed `openapi.json` snapshot.

## Refreshing from the live backend

The SpringDoc `/v3/api-docs` endpoint is **auth-protected** (it sits behind the
`anyRequest().authenticated()` rule), so generation runs from a committed
snapshot — not a live unauthenticated fetch. **Run these steps only against a
local dev instance.**

```bash
# 1. Start the backend in the DEV profile (api-docs is disabled in prod).
#    There is no unix `mvnw` in this repo — use a global maven. The dev profile
#    needs the usual env (DATABASE_URL, TOKEN_SECRET, MAIL_HOST, ...; see envExample).
cd Beyou-backend-spring && mvn spring-boot:run -Dspring-boot.run.profiles=dev

# 2. Log in as a VERIFIED user. The access JWT comes back in the
#    `X-Access-Token` RESPONSE HEADER (not the body). A fresh user must have
#    email_verified=true (the dev seeder does not create test@test.com).
TOKEN=$(curl -s -D - -o /dev/null -X POST http://localhost:8099/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"<your-dev-user>","password":"<password>"}' \
  | grep -i '^x-access-token:' | sed 's/.*: //I' | tr -d '\r')

# 3. Fetch the spec and write it PRETTY-PRINTED (readable git diffs on refresh)
curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:8099/api/v1/v3/api-docs \
  | node -e "process.stdin.resume();let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>process.stdout.write(JSON.stringify(JSON.parse(d),null,2)+'\n'))" \
  > packages/contracts/openapi.json

# 4. Regenerate TypeScript types and verify
npm --workspace @beyou/contracts run generate
npm --workspace @beyou/contracts run typecheck
```

## Scripts

| Script | Purpose |
|--------|---------|
| `generate` | Regenerate `src/schema.d.ts` from `openapi.json` |
| `check` | Drift gate — fails if `schema.d.ts` is stale vs `openapi.json` |
| `typecheck` | Compile-time validation of all types including `assert.ts` guard |

The `check` script is the drift gate: it regenerates types to a temp file and diffs against the committed `schema.d.ts`. It catches the case where `openapi.json` was updated but `schema.d.ts` was not regenerated.

> **Note:** `check` only validates that `schema.d.ts` matches the committed `openapi.json`. It does **not** detect drift between the *live backend* and this snapshot (e.g. the backend added an endpoint and nobody refreshed `openapi.json`). A separate CI job that boots the stack, fetches the live spec, and diffs it against `openapi.json` is planned (Phase 0 Task 9 / e2e pipeline). Until then, refresh the snapshot manually after backend API changes.
